# Easy Analysis Skill

## Purpose

对复杂技术文档进行逐段精读、翻译、总结，确保彻底理解每个细节，特别适合英语不熟练的用户深入理解英文技术文档。

## When to Use

- 分析 skill 定义文件 (SKILL.md)
- 理解复杂的工作流程文档
- 学习第三方框架/工具的设计思想
- 需要「复刻」某个功能前，彻底理解其实现
- 用户明确表示"看不懂"或"理解困难"时

## Analysis Method

### Step 1: Paragraph-by-Paragraph Reading

**必须逐段处理**，每段原文按以下格式输出：

```markdown
### 段落 N

**原文:**
[原文内容]

**翻译:**
[中文翻译，尽量直译但保持通顺]

**要点:**
- 要点 1
- 要点 2
- 要点 3
```

### Step 2: Reference File Analysis

如果段落引用了外部文件，在段落后追加详细解释：

#### 对于脚本文件:
- **代码结构**: 分段解释，每段加注释
- **关键逻辑**: 核心算法/流程说明
- **数据流**: 输入输出、状态变化

#### 对于文档文件:
- **结构概述**: 文件整体结构
- **核心概念**: 定义的关键概念
- **使用示例**: 提供的示例代码

### Step 3: Final Summary

所有段落分析完成后，提供：

```markdown
## 整体总结

### 核心概念
[文档中定义的关键术语和概念]

### 工作流程
[步骤化的流程图/清单]

### 关键文件
[文件清单及其作用]

### 如何复刻/应用
[实际应用该 skill 的方法]
```

## Output Format

分析结果保存到 `docs/<project-name>/` 目录：

```
<NN>-<skill-name>-analysis.md
```

例如：`07-brainstorming-skill.md`

## Rules

1. **每段都翻译** - 即使看起来简单的段落，也必须提供中文翻译
2. **保持结构层次** - 严格跟随原文的标题层级
3. **引用文件详细解释** - 让读者无需再看原文件
4. **使用中文** - 所有解释、总结、要点都必须用中文
5. **格式一致** - 严格按照"原文→翻译→要点"的顺序

## Example

用户要求："分析一下 @docs/gsd/01-map-codebase-workflow.md"

输出结构：
- 段落 1: 元数据 (原文/翻译/要点)
- 段落 2: 流程概述 (原文/翻译/要点)
- 段落 3: 步骤详解 (原文/翻译/要点)
- ...
- 引用文件解释 (如有)
- 整体总结
